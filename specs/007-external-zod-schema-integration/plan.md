# Implementation Plan: External Zod Schema Integration

**Branch**: `007-external-zod-schema-integration` | **Date**: 2026-04-03 | **Spec**: [spec.md](./spec.md)
**Input**: Feature spec from `/specs/007-external-zod-schema-integration/spec.md`

## Summary

Refactor the CLI's ES API command infrastructure to support a unified Zod schema where path, query, and body parameters are all top-level fields with `found_in` metadata (via Zod v4's `.meta(...)`). This replaces the current `EsApiDefinition` structure that uses separate `pathParams[]`, `queryParams[]`, and `body` fields. The new pattern is required to consume externally generated schemas from `@elastic/zod` and will be adopted by all existing hand-authored commands.

## Technical Context

**Language/Version**: TypeScript (via Node.js `--experimental-strip-types`), Node.js 22+
**Primary Dependencies**: Commander.js (CLI framework), Zod v4 (validation), `@elastic/transport` (ES HTTP transport), `@elastic/zod` (external, schemas only)
**Storage**: N/A (CLI tool — reads config from YAML files)
**Testing**: Node.js built-in test runner (`node:test`)
**Target Platform**: Windows, Linux, macOS (per Constitution VII)
**Project Type**: CLI
**Performance Goals**: Command registration ≤10% overhead vs. baseline (SC-003)
**Constraints**: Minimal dependencies (Constitution VI), strict TypeScript, no build step
**Scale/Scope**: ~50+ existing hand-authored ES API commands to migrate; eventually hundreds generated by external tooling
